# Фронтенд

## Обзор

React 19 + TypeScript + Vite, CSS Modules. Без Redux — глобальное состояние живёт в Zustand-сторах (`shared/api/*Store.ts`) и React Context (`shared/context/`), локальное — в обычном `useState`/кастомных хуках.

**Структура папок:**

```
frontend/src/
├── app/router/        — маршрутизация, guard-层 (авторизован? есть профиль?)
├── pages/              — страницы, в т.ч. MessengerPage со своими hooks/
├── widgets/            — крупные составные блоки (ChatWindow, ChatListPanel)
├── features/           — модалки и формы конкретных фич (профиль, группы)
├── shared/
│   ├── api/             — HTTP-клиенты, SignalR-клиент, Zustand-сторы
│   ├── context/          — React Context (тема, профиль, фичефлаги)
│   ├── hooks/            — переиспользуемые хуки без домена
│   ├── lib/               — чистые утилиты (даты, файлы, auth-токены)
│   ├── types/             — общие типы
│   └── ui/                — низкоуровневые UI-примитивы
```

Серверные модули и их REST/WebSocket-контракты описаны в `docs/modules/*.md` — здесь только то, что специфично для клиента: как состояние синхронизируется, кэшируется и переживает переключение чатов.

## Аутентификация и кросс-вкладочная синхронизация

`shared/lib/auth/authTokens.ts` хранит JWT в `localStorage` и рассылает кросс-вкладочные события через `'storage'` — это нативное DOM-событие, которое браузер шлёт во **все другие** вкладки того же origin при изменении `localStorage` (во вкладке, где вызван `clearAuthTokens()`/`saveAuthTokens()`, событие не всплывает — она обновляет свой стейт напрямую, отдельным вызовом).

`App.tsx` подписывается на оба случая:
- **Логаут** (`onAuthTokensCleared`) — кнопка "Выйти" в одной вкладке или неудачный refresh в `apiClient` должны разлогинить и остальные открытые вкладки. Используется полная перезагрузка на `/login` (`window.location.href`), а не редирект внутри SPA — иначе SignalR-соединение, закэшированные чаты и профиль остались бы висеть в памяти остальных вкладок.
- **Логин** (`onAuthTokensSaved`) — вход/регистрация/смена аккаунта в одной вкладке должны обновить остальные. Здесь достаточно `window.location.reload()`: `GuardedLayout` (`app/router/AppRouter.tsx`) сам решит, куда вести по актуальным токенам.

Декодирование JWT в `authTokens.ts` пробует claim `sub` и запасной вариант `"nameid"` (полный URI `ClaimTypes.NameIdentifier` тоже) — см. [architecture.md — Авторизация](architecture.md) про то, почему бэкенд может отдать оба варианта в зависимости от версии .NET.

## Realtime: SignalR

`shared/api/signalrClient.ts` — синглтон-обёртка (`export const signalR = new SignalRClient()`) над `@microsoft/signalr`. Реконнект — `withAutomaticReconnect` с задержками `0s, 2s, 10s, 30s`, дальше держит последнюю. Каждое серверное событие оборачивается в `on<Event>(handler) → () => void` (отписка) — токен передаётся через query string при подключении, поскольку браузерный WebSocket API не поддерживает кастомные заголовки при upgrade-запросе.

`shared/api/useSignalR.ts` — React-хук поверх клиента:
- Группу чата **не покидает** при смене/размонтировании компонента — соединение держит подписку на все чаты пользователя постоянно (см. ниже); `joinChat` внутри хука — просто подстраховка для чата, которого ещё нет в сторе.
- Колбэки-обработчики читают актуальные значения из `useRef`, а не из замыкания рендера, в котором были переданы: без этого новый объект `options` на каждый рендер пересоздавал бы подписки (`off()` затем `on()`), и в окне между ними реальное событие могло бы потеряться.

### Вступление во все чаты и двойная доставка

`ConnectedLayout` (`app/router/AppRouter.tsx`) при подключении вступает во **все** SignalR-группы чатов пользователя разом (ждёт `chatsLoaded`, чтобы не слать `joinChat` с ещё не загруженными не-Guid идентификаторами черновиков). При изменении набора чатов (новый чат появился/пропал) состав групп синхронизируется тем же эффектом.

После реконнекта список чатов перезапрашивается явно (`loadChats()`) — SignalR не досылает пропущенные во время обрыва broadcast-события, а список чатов мог устареть. На самом первом коннекте это не нужно — он уже покрыт начальной загрузкой.

Сервер шлёт события и в группу чата (`chat:{id}`), и в личную группу участника — если получатель уже состоит в обеих (обычное дело, раз клиент вступает во все чаты сразу), одно и то же событие приходит **дважды**. Дедупликация — на клиенте, по `messageId`/по факту уже применённого изменения: `useChatMessages.ts → handleIncomingMessage` и `chatsStore.ts → handleNewMessage` перед применением проверяют, не обработано ли уже такое событие.

### Индикатор набора текста

`pages/MessengerPage/hooks/useTypingIndicator.ts`: `StartTyping` шлётся на **каждое** нажатие клавиши, а не только при первом — если слать только при первом, то после паузы (когда собственный debounce-таймер уже сбросился) следующее нажатие ссылалось бы на устаревшее замыкание состояния. `StopTyping` — по debounce (2с) или явно при отправке; серверный таймаут на чужой стороне (3с) — подстраховка на случай, если `StopTyping` не дойдёт (обрыв связи).

## Список чатов — `chatsStore`

Zustand-стор (`shared/api/chatsStore.ts`), инвариант: **сервер — источник истины** для `unreadCount`, кроме одного случая — чат, открытый прямо сейчас в этой вкладке (`activeChatId`). Пока `POST /chats/{id}/read` (см. [Chats — Прочитано](modules/chats.md#прочитано-read-receipts-реальное-время)) не долетел до сервера, ответ `loadChats()` ещё может вернуть старый `unreadCount` — стор обнуляет unread для активного чата поверх серверного значения, чтобы счётчик не "моргал" обратно.

`loadChats()` схлопывает конкурентные вызовы в один запрос через модульную переменную `loadChatsInFlight` — несколько компонентов могут запросить перезагрузку списка одновременно (реконнект, входящее сообщение в неизвестный чат, ручной retry), но летит один HTTP-запрос.

`handleNewMessage`: если чата с этим `chatId` ещё нет в сторе (типичное дело для первого сообщения новому собеседнику — см. [Chats — фронтенд без пустых чатов](modules/chats.md#фронтенд-поиск-пользователя-и-создание-чата-без-пустых-чатов)), стор дотягивает список целиком через `loadChats()`, а не пытается сконструировать запись локально. Свои же сообщения (включая пересланные) никогда не увеличивают unread, даже если чат не открыт.

`handleMessagesRead`: `otherReadAt` обновляется, только если `readerId` — не текущий пользователь (событие о собственном прочтении с другого устройства не про статус "прочитано собеседником"), и только если новое значение позже уже сохранённого — события от разных участников группы могут прийти не по порядку, устаревшее не должно откатить галочку назад.

## Сообщения и история — `useChatMessages`

`pages/MessengerPage/hooks/useChatMessages.ts` — самый плотный хук в проекте, держит состояние сообщений по всем открытым за сессию чатам (`Record<chatId, Message[]>`), не только по текущему — поэтому переключение между уже открытыми чатами не требует повторной загрузки истории (см. также [Files — кэш вложений](#вложения-и-файлы) — та же идея на уровне файлов).

- **Дедуп сообщений.** `messages` — не сырой массив, а `useMemo`-фильтр по `messageId`: гонка между optimistic-вставкой при отправке, догрузкой истории и повторной realtime-доставкой (см. выше) может дать два элемента с одним `messageId` — остаётся первый.
- **Эхо собственных сообщений.** `handleIncomingMessage` игнорирует своё же сообщение, если оно уже показано локально через optimistic UI при `send()` — кроме пересланных копий и системных событий, у которых нет локального оптимистичного добавления, они должны появиться именно по этому realtime-событию.
- **Отправка текста** (`send`) — сразу вставляет сообщение локально со статусом `pending`, реальный запрос (`doSend`) идёт через SignalR; при успехе статус меняется на `sent` с настоящим `messageId`, при ошибке — на `failed` (доступен `retry`).
- **Отправка файлов** (`sendFiles`) — только REST (`POST /messages/upload`), не SignalR: та не годится для бинарных данных. Здесь нет пути `pending → sent` — bubble сразу приходит с готовым результатом, а realtime-эхо (тот же файл, пришедший по WebSocket) дедуплицируется по `messageId` тем же механизмом, что и обычные сообщения. На черновом чате (см. ниже) история могла подгрузиться с этим же сообщением раньше, чем отработает колбэк отправки — порядок между двумя независимыми async-цепочками не гарантирован, поэтому дедуп обязателен и здесь.
- **Курсорная пагинация истории** (`loadMoreHistory`) — если целая подгруженная страница оказалась полностью удалёнными сообщениями (см. [Messages — мягкое удаление](modules/messages.md#удаление-сообщения-мягкое)), `IntersectionObserver` больше не сработает: страница дала 0 видимых элементов. `fetchUntilVisible` рекурсивно тянет страницы подряд, пока не найдёт хотя бы одно неудалённое сообщение.
- **Правки цитат.** И удаление, и редактирование сообщения обновляют не только само сообщение, но и все **чужие** сообщения, чья `replyToContent`/цитата на него ссылается — цитата в этом мессенджере "живая" (см. [Messages — ответ на сообщение](modules/messages.md#ответ-на-сообщение-reply)), а не замороженная копия текста на момент ответа.
- **Обновление профиля.** `handleUserProfileUpdated` патчит имя/аватар/цвет отправителя во **всех** закэшированных чатах, не только открытом сейчас — иначе сообщения в свёрнутых чатах остались бы со старыми денормализованными данными до следующей полной перезагрузки.

`REPLY_PREVIEW_MAX_LENGTH = 120` должно совпадать с `MessagePreview.MaxLength` на бэкенде ([Messages](modules/messages.md)) — превью цитаты живое (обновляется при правке оригинала) и обрезается тем же образом что при отправке, что при загрузке истории.

## Поиск по чату и переход к сообщению

`ChatSearchPanel.tsx` — самодостаточная панель (свой debounce, свой запрос), открывается кнопкой-лупой в шапке `ChatWindow.tsx`. Сложность не в самой панели, а в переходе к найденному: результат поиска может быть сообщением, ещё не подгруженным на клиент (поиск идёт по всей истории на сервере — см. [Messages — поиск по сообщениям](modules/messages.md#поиск-по-сообщениям)), а `ChatWindow` умеет скроллить только к тому, что уже есть в DOM.

Разделение ответственности между компонентами:
- `MessengerPage.tsx` держит состояние `jumpTarget` — по клику на результат поиска запускает эффект, который проверяет, есть ли уже сообщение с этим `messageId` среди загруженных, и если нет и история ещё не кончилась — дозагружает следующую страницу тем же `loadMoreHistory`, что и обычная бесконечная прокрутка вверх. Эффект перезапускается при каждом изменении `messages`/`historyLoaded`, так что дозагрузка продолжается постранично, пока сообщение не найдётся или не кончится история.
- `ChatWindow.tsx` получает `jumpToMessageId` как проп и в своём эффекте просто проверяет, появился ли элемент с нужным `data-message-id` в `messagesRef` — как только да, скроллит и подсвечивает (`scrollToMessage`, тот же механизм, что уже используется для перехода по цитате в ответе).

Разделение сознательное: `MessengerPage` владеет загрузкой истории и не знает про DOM, `ChatWindow` владеет DOM и не знает про пагинацию — ни одному компоненту не нужно доступа к внутренностям другого.

## Опросы — `PollMessage.tsx` / `CreatePollModal.tsx`

Основная модель (кто что видит, реалтайм, только группы) — в [Messages — Опросы](modules/messages.md#опросы). Здесь — то, что специфично для клиента:

- **Собственный голос не обновляется оптимистично.** `votePoll`/`retractPollVote` (`useChatMessages.ts`) просто дёргают REST и ничего не трогают в локальном состоянии — экран обновляется только когда придёт `PollVoteChanged` по WebSocket (тот же принцип, что и у реакций, см. [Realtime](modules/realtime.md)). Раз событие уходит и на собственные соединения тоже, лишний код для "предположим, что получится" не нужен.
- **Список проголосовавших** сворачиваемый (`expandedOptionId` в `PollMessage.tsx`) — по умолчанию видны только аватарки (до 5) и счётчик, а разворачивание по клику показывает полные имена; клик по имени открывает профиль пользователя тем же колбэком, что уже используется для клика по автору пересланного сообщения (`onForwardedUserClick`, прокинут в `PollMessage` под именем `onVoterClick`).
- **Меню кнопки-скрепки** (`ChatWindow.tsx`) показывается по наведению только в группах и закрывается с задержкой ~150мс после ухода курсора (`closeAttachMenu`, тот же паттерн отложенного закрытия, что уже использовался для превью "кто поставил реакцию" при наведении, `closeReactionPreview`) — без задержки курсор успевал выйти из кнопки раньше, чем добраться до самого меню.
- **Фокус после выбора файлов.** Выбор файлов через системный диалог оставляет фокус на кнопке-скрепке, а не на поле ввода (`<input type="file">` скрыт через `display:none` и в принципе не может быть в фокусе) — нажатие Enter в этот момент срабатывало как повторный клик по кнопке вместо отправки сообщения. Фикс — явный `focus()` на textarea сразу после `handleFileSelect`.

## Скролл истории — `useScrollRestore`

`pages/MessengerPage/hooks/useScrollRestore.ts` управляет двумя независимыми сигналами:

- `bottomSignal` — явный запрос "прокрутить в низ" (`scrollToBottomNow`), дёргается при открытии чата и при отправке/получении сообщения.
- `restoreSignal` — после подгрузки более старой истории вверх компенсирует прыжок скролла (сохраняет `scrollHeight`/`scrollTop` до вставки, восстанавливает разницу после).

Эффект, слушающий `bottomSignal`, **намеренно** не в зависимостях от списка сообщений — здесь скрывался баг: если добавить `messages` в зависимости (чтобы эффект видел уже отрисованный DOM), то после первого срабатывания `bottomSignal` (он остаётся ненулевым до конца жизни компонента) **любое** последующее изменение `messages` — например, простановка реакции на старое сообщение при чтении истории — заново гоняло эффект и утаскивало вид в низ чата, даже когда пользователь читает историю выше. Работает и без `messages` в зависимостях: `setChatMessages` и `scrollToBottomNow` в `useChatMessages.ts` всегда вызываются синхронно друг за другом в одной функции, React 18 батчит их в один коммит, так что к моменту, когда `bottomSignal` меняется, `messagesRef` уже отражает обновлённый DOM.

## Вложения и файлы

### Кэш blob-URL — `fileBlobCache.ts`

Вложения чата (в отличие от аватарок) отдаются не анонимно — бэкенд проверяет членство в чате (см. [Files — приватность](modules/files.md#приватность-при-скачивании)), поэтому обычный `<img src="/api/files/...">` получил бы `401` (браузер не шлёт `Authorization` для `<img>`). `useAuthedFileUrl` качает файл вручную с токеном и превращает в blob-URL.

`ChatWindow` пересоздаётся целиком при переключении чата (`key={chatId}` в `MessengerPage.tsx`) — без общего кэша это означало бы повторную загрузку тех же картинок при каждом возврате в уже открытый чат. `fileBlobCache.ts` — модульный (не привязанный к React-дереву) кэш поверх `fileUrl`:

- `acquireFileBlobUrl`/`releaseFileBlobUrl` — реф-каунт; повторный `acquire` того же `fileUrl`, пока загрузка ещё идёт, подписывается на тот же промис и тот же прогресс, а не запускает вторую загрузку.
- Ограничение на 150 одновременно живых blob-URL (каждый держит декодированные байты файла в памяти вкладки) — при превышении вытесняются самые давно отпущенные (`lastReleasedAt`), а не только что использованные.
- `null` в колбэке прогресса означает "сервер не прислал `Content-Length`" (см. [Files — кэширование](modules/files.md#кэширование)), а не "загрузка не началась".
- Кэш целиком очищается при логауте (`clearFileBlobCache`, вызывается из `IconNav.tsx`) — иначе данные вышедшего пользователя остаются в памяти вкладки.

### Очередь вложений при отправке — `useAttachmentQueue.ts`

- Проверка всех выбранных файлов идёт разом при добавлении в очередь: если хоть один не проходит (тип/размер), в очередь не добавляется **ни один** — иначе можно случайно отправить часть альбома, не заметив, что один файл отсеялся.
- `ChatWindow` размонтируется целиком при смене чата (`key={chatId}`) — если пользователь выбрал файлы, но не отправил и переключился на другой чат, blob-превью выбранных файлов нужно освобождать явно в cleanup, иначе они никогда не освободятся.
- Подсчёт "глубины" `dragenter`/`dragleave` вместо однократного флага — эти события всплывают с дочерних элементов, и оверлей drag-and-drop мигал бы при движении курсора над дочерними узлами без этого счётчика.
- Отправка всей очереди — один HTTP-запрос (одно сообщение с несколькими вложениями), не по файлу за раз; поэтому и прогресс-бар (`FilePreviewBar.tsx`) общий на весь запрос, а не индивидуальный по файлу.

### Разрешённые типы — `fileType.ts`

`ALLOWED_ATTACHMENT_EXTENSIONS`/`isAllowedAttachment` и `MAX_ATTACHMENT_SIZE_BYTES` (25 МБ) должны совпадать с `AllowedAttachmentMimeTypes` на бэкенде ([Files — ограничения](modules/files.md#ограничения)) — это клиентская проверка для мгновенной обратной связи, а не единственная линия защиты (сервер валидирует независимо, вплоть до сигнатуры файла). `isAllowedAttachment` явно исключает `image/svg+xml` из образов — как и на сервере, SVG может нести `<script>`.

Определение типа файла для проверки идёт по **расширению имени**, а не по `File.type` — на некоторых ОС/браузерах MIME для `.rar`/`.7z`/`.docx` и т.п. просто не определяется. Если расширения нет или оно незнакомое, тогда уже доверяем MIME, если браузер его распознал.

Иконки типов файлов (`FileTypeIcon.tsx`/`getFileTypeInfo`) используют ту же цветовую палитру, что и аватарки (`AVATAR_COLORS`) — чтобы цветные бейджи файлов не выглядели чужеродно рядом с остальным интерфейсом.

## Профиль и аватарка

`ProfileSetupForm.tsx`/`EditProfileModal.tsx` дублируют одну и ту же логику (существует и переиспользуемый хук `shared/hooks/useAvatarCrop.ts` с тем же поведением):

- Размер файла проверяется **до** обрезки, не после — сама обрезка (`canvas.toBlob`) пересжимает и обычно уменьшает размер, так что проверка только итогового файла пропустила бы изначально огромный файл, если кроп-область оказалась маленькой.
- При сабмите формы ни один запрос не уходит, пока хоть одно поле невалидно, включая необязательные (город, отдел, статус, телефон) — иначе бэкенд может успеть сохранить часть полей до того, как найдёт первое невалидное, и форма с сервером разойдутся во мнении, что сохранено.
- `ProfileSetupForm`: профиль создаётся одним атомарным запросом со всеми полями сразу — `409 ProfileAlreadyExists` в ответ означает, что он уже был создан в прошлой попытке (например, запрос ушёл, но клиент не дождался ответа), и это не должно блокировать переход дальше.

`useAvatarCrop.ts`: ref'ы для preview/crop-URL нужны конкретно в cleanup-эффекте на размонтирование (`useEffect(() => () => {...}, [])`) — с пустым списком зависимостей замыкание внутри cleanup видело бы только значения из первого рендера, а не актуальные URL для `URL.revokeObjectURL`.

## Прочие детали

- **`formatDateTime.ts`.** Время сообщения — всегда 24-часовой формат, не зависящий от языка интерфейса (смена языка — это перевод слов, а не смена системы счисления времени суток). Метка-разделитель дат ("Сегодня"/"Вчера"/дата) пересчитывается при каждом рендере, чтобы обновляться при смене языка без перезагрузки чата — но группировка сообщений по дням (`MessageList.tsx → buildRenderedItems`) использует отдельный языконезависимый `dateKey`, а не переведённую метку: иначе смена языка развела бы сообщения одного дня по разным группам (`"Today"` vs `"Сегодня"` как разные ключи).
- **`MessageList.tsx`.** Системные сообщения (о смене состава группы) не считаются "соседями" при группировке аватарок/имени соседних сообщений — при определении `showAvatar`/`showName` ищется ближайшее **не системное** сообщение по обе стороны, а не буквально `messages[i ± 1]`.
- **`useUserSearch.ts` / `useGroupMembers.ts`.** Оба защищены от гонки устаревших ответов одним и тем же паттерном (ref с "последним запрошенным" id/запросом) — если более старый запрос (например, поиск `"al"`) отвечает позже более нового (`"alex"`), его результат игнорируется, а не затирает актуальный.
- **`useGroupMembers.ts`.** При обновлении списка участников порядок отображения намеренно сохраняется — существующие патчатся на месте, новые дописываются в конец, удалённые выпадают, чтобы смена роли участника не переставляла всех местами в списке.
- **`ChatWindow.tsx`.** Клиентский лимит длины сообщения должен совпадать с лимитом на бэкенде (`Message.Create`/`Message.Edit`) — проверяется до отправки, чтобы слишком длинное сообщение не превращалось молча в статус "не отправлено". Отдельно есть чисто клиентское ограничение на число строк (бэкенд их не считает) — без него вставка сотен коротких строк давала бы сообщение, укладывающееся в лимит символов, но нечитаемо длинное.
- **`toastStore.ts`/`Toast.tsx`.** Единственный оставшийся канал мгновенной обратной связи об неудачном действии (удаление сообщения, выход из группы и т.п.) — модальный `ErrorModal` для этого класса ошибок убрали в пользу браузерных push-уведомлений, но те для сиюминутных ошибок действий не подходят (не должны уходить в системный центр уведомлений).
- **`browserNotifications.ts`.** При отписке от push использует `getRegistration()`, а не `navigator.serviceWorker.ready` — `.ready` никогда не резолвится, если воркер не был зарегистрирован вообще, что подвесило бы логаут навсегда вместо простого пропуска несуществующей подписки.
- **`ConnectionBanner.tsx`.** Баннер "Подключено" показывается только после реального восстановления после обрыва — если соединение раньше не рвалось (обычный первый коннект при загрузке страницы), баннер не нужен.
