# Модуль Auth — аутентификация

## Что делает

Управляет учётными данными пользователей: регистрация, вход, выход, обновление токена.

## Схема БД

Схема `auth`.

### Таблица `auth.user`

| Колонка | Тип | Описание |
|---|---|---|
| id | uuid | Первичный ключ |
| email | varchar(255) | Email (уникальный) |
| password_hash | varchar(512) | Хеш пароля (Argon2) |
| is_email_verified | boolean | Подтверждён ли email |
| created_at | timestamptz | Дата регистрации |

**Индексы:** `ix_user_email` (unique) — быстрый поиск по email при логине.

### Таблица `auth.refresh_token`

| Колонка | Тип | Описание |
|---|---|---|
| id | uuid | Первичный ключ |
| user_id | uuid | Ссылка на auth.user |
| token | varchar(256) | **SHA-256 хеш** токена (уникальный), не сам токен — при утечке БД значения в этой колонке нельзя использовать напрямую как рабочий refresh-токен |
| expires_at | timestamptz | Когда истекает |
| created_at | timestamptz | Когда создан |
| is_revoked | boolean | Отозван ли (logout) |

**Индексы:** `ix_refresh_token_token` (unique), `ix_refresh_token_user_id`.

Сам токен — 64 случайных байта (`JwtTokenService.GenerateRefreshToken`), уже достаточно высокоэнтропийный: в отличие от паролей, здесь не нужен медленный KDF (Argon2/bcrypt) для защиты от подбора, быстрого SHA-256 достаточно (`RefreshToken.Hash`).

## API endpoints

| Метод | URL | Описание |
|---|---|---|
| GET | `/api/auth/features` | Публичные фичефлаги (`twoFactorEnabled`, доступность сброса пароля) — фронтенд скрывает недоступные поля формы |
| POST | `/api/auth/register` | Регистрация нового пользователя |
| POST | `/api/auth/login` | Вход. Если 2FA включена — `{ requiresOtp: true }` вместо токенов |
| POST | `/api/auth/verify-otp` | Подтверждение 6-значного кода из письма, выдаёт токены |
| POST | `/api/auth/refresh` | Обновление JWT по refresh-токену |
| POST | `/api/auth/logout` | Выход (отзыв refresh-токена) |
| POST | `/api/auth/forgot-password` | Запрос кода сброса пароля на email |
| POST | `/api/auth/reset-password` | Сброс пароля по коду |

## Как работает логин

1. Клиент отправляет email + password
2. Система находит пользователя в `auth.user` по email
3. Сверяет пароль с хешем через **Argon2** (алгоритм безопасного хеширования)
4. Генерирует **JWT Access Token** (живёт 15 минут) — подписанный токен с userId внутри
5. Генерирует **Refresh Token** (живёт 7 дней) — длинная случайная строка, в БД хранится только её SHA-256 хеш (см. схему выше)
6. `accessToken` возвращается в теле JSON-ответа. `refreshToken` в тело ответа **не попадает** (`[JsonIgnore]` на DTO) — он кладётся только в httpOnly-cookie (`RefreshTokenCookie`, имя `messenger_refresh_token`, путь `/api/auth`, `Secure`+`SameSite=None` вне Development). Так refresh-токен недоступен из JS/девтулс на клиенте, даже там, где `httpOnly` как раз и должен был это предотвратить — если бы он дублировался в JSON-теле, это защиту сводило бы на нет

## Email-подтверждение, 2FA и сброс пароля

Обе фичи — за флагами в конфиге (`TwoFactor:Enabled`, `PasswordReset:Enabled`; в Docker пробрасываются из `.env` через `TWO_FACTOR_ENABLED`/`PASSWORD_RESET_ENABLED`, см. `docker-compose.yml`) и требуют настроенного `Resend` (`RESEND_API_KEY`) — без него письма с кодом просто не уйдут, даже если флаг включён.

- **2FA при регистрации/логине.** Если `TwoFactor:Enabled = true`, `register`/`login` не возвращают токены сразу, а шлют 6-значный код на email и отвечают `{ requiresOtp: true }`. Код живёт 5 минут в `IMemoryCache` (не в БД — короткоживущий одноразовый секрет), клиент подтверждает его через `POST /api/auth/verify-otp`, который и выдаёт `TokenPairDto`.
- **Сброс пароля.** `POST /api/auth/forgot-password` **всегда** возвращает `200`, даже если email не найден в базе — так эндпоинт не даёт узнать, зарегистрирован ли конкретный адрес в системе (перечисление аккаунтов). Код сброса — та же схема: `IMemoryCache`, 5 минут.
- Оба эндпоинта, где подбирается короткий числовой код (`verify-otp`, `forgot-password`, `reset-password`), — под политикой `auth-strict` (см. «Rate limiting» ниже), а не обычной `auth`.

## Как работает авторизация последующих запросов

Все защищённые эндпоинты требуют заголовок:
```
Authorization: Bearer <jwt_access_token>
```

JWT декодируется, из него извлекается `userId` — он используется в хендлерах.

**Проверка после декодирования подписи.** Валидная подпись и срок годности — не единственное условие: `OnTokenValidated` дополнительно смотрит, существует ли пользователь с этим `userId` в БД. Без этой проверки токен пересозданного/удалённого пользователя (типичный dev-кейс — база пересоздана, а в браузере остался старый access token) формально проходил бы проверку подписи, но затем упирался бы в `404` на каждом отдельном запросе — а не в явный `401`, из-за чего клиент никогда не попытался бы обновить или сбросить его сам. Такой токен роняется явно на уровне аутентификации: клиент получает обычный `401`, пробует `/auth/refresh`, тот тоже не находит пользователя и стирает и access, и refresh-cookie.

## Rate limiting

Чувствительные auth-эндпоинты троттлятся по IP (`RateLimitingExtensions`, fixed window) — без этого `login`/`verify-otp`/`reset-password` можно долбить перебором без каких-либо ограничений (пароль, 6-значный OTP-код, код сброса пароля):

| Политика | Лимит | Эндпоинты |
|---|---|---|
| `auth` | 10 запросов / минуту / IP | `register`, `login`, `refresh` |
| `auth-strict` | 5 запросов / 5 минут / IP | `verify-otp`, `forgot-password`, `reset-password` — узкое окно для эндпоинтов, где подбирается короткий числовой секрет (OTP, код сброса) |

## Refresh Token Flow

```
JWT истёк → клиент вызывает POST /api/auth/refresh (тело можно не слать — сервер сам
  читает refresh token из httpOnly-cookie, см. выше; JSON-тело — фолбэк для не-браузерных клиентов) →
  сервер хеширует полученный токен и ищет совпадение в БД →
  выдаёт новый JWT + новый refresh token (тоже только через cookie) →
  старый refresh token отзывается (is_revoked = true)
```

Если refresh-токен из cookie невалиден (истёк, отозван, либо — типичное дело в dev — база пересоздана и такой строки в `auth.refresh_token` больше нет), сервер явно удаляет саму cookie (`RefreshTokenCookie.Delete`). Без этого шага она осталась бы в браузере навсегда (клиент не может стереть httpOnly-cookie через JS) и каждый следующий `/refresh` падал бы с той же ошибкой, пока пользователь не почистит cookies вручную.

## Связь с модулем Users

Auth знает только "кто может войти", Users знает "как пользователь выглядит" — это намеренное разделение (см. [Users](users.md)). После регистрации клиент сам вызывает `POST /api/users`, чтобы создать профиль; `auth_user_id` в `user_profile` связывает `auth.user` и `users.user_profile` без FK между схемами.
