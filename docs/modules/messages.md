# Модуль Messages — сообщения

## Что делает

Хранит все сообщения, управляет их жизненным циклом. При сохранении нового/изменённого сообщения — публикует **доменное событие**, которое подхватывает Realtime-модуль и рассылает по WebSocket (см. [Realtime](realtime.md)).

## Схема БД

Схема `messages`.

### Таблица `messages.message`

| Колонка | Тип | Описание |
|---|---|---|
| id | uuid | Первичный ключ |
| sequence | bigint | Автоинкрементный identity — монотонный тай-брейкер к `sent_at` для курсорной пагинации (см. ниже); сам по себе клиенту не отдаётся |
| chat_id | uuid | Какой чат (FK на chats.chats) |
| sender_id | uuid | Кто отправил (FK на auth.user) |
| content | varchar(4096) | Текст сообщения |
| status | varchar(20) | `"Sent"`, `"Delivered"`, `"Read"`, `"Deleted"` |
| sent_at | timestamptz | Когда отправлено |
| edited_at | timestamptz | Когда отредактировано (null если нет) |
| deleted_at | timestamptz | Когда удалено (null если нет) |
| reply_to_message_id | uuid | Ответ на какое сообщение (FK на messages.message, SET NULL при удалении оригинала) |
| forwarded_from_message_id | uuid | Если сообщение — пересланная копия: id исходного сообщения (FK на messages.message, SET NULL если оригинал удалён) |
| forwarded_from_user_id | uuid | Автор исходного (пересылаемого) сообщения — для подписи «Переслано от», не совпадает с `sender_id` (FK на auth.user, SET NULL) |
| message_type | varchar(10) | `"Text"` (обычное), `"System"` — системное сообщение об изменении состава группы (см. «Системные сообщения» ниже) или `"Poll"` — опрос (см. «Опросы» ниже) |
| system_event_type | varchar(20) | Заполнено только для `message_type = "System"`: `"MemberAdded"` \| `"MemberLeft"` \| `"MemberRemoved"` |
| target_user_id | uuid | Заполнено только для `message_type = "System"` — кого именно добавили/удалили/кто вышел. `sender_id` при этом — кто выполнил действие (сам ушедший для `MemberLeft`, админ для `MemberAdded`/`MemberRemoved`) |

Вложений на `message` нет отдельной колонкой — одно сообщение может нести несколько файлов сразу, они живут в отдельной таблице `messages.message_attachment` (ниже).

**Индексы:**
- `ix_message_chat_id_sent_at` — пагинация истории чата (самый частый запрос)
- `ix_message_chat_id_sequence` — курсорная пагинация (см. «Cursor-пагинация» ниже)
- `ix_message_sender_id` — поиск сообщений по отправителю
- `uq_message_sequence` (unique) — гарантия, что `sequence` действительно уникален и монотонен

**Concurrency:** редактирование/удаление защищено optimistic concurrency через системную колонку Postgres `xmin` (не требует отдельной миграции — есть у каждой строки "из коробки"). Если сообщение успели изменить/удалить параллельно между чтением и записью — `EditMessageCommandHandler`/`DeleteMessageCommandHandler` ловят `DbUpdateConcurrencyException` и возвращают `409 Conflict` вместо тихой перезаписи.

### Таблица `messages.message_attachment`

| Колонка | Тип | Описание |
|---|---|---|
| id | uuid | Первичный ключ |
| message_id | uuid | Владеющее сообщение (FK на messages.message, CASCADE) |
| file_url | varchar(2048) | URL файла в хранилище |
| file_name | varchar(255) | Оригинальное имя файла |
| content_type | varchar(100) | MIME-тип |

**Индекс:** `ix_message_attachment_message_id`. **Ограничение:** максимум 10 вложений на сообщение (`Message.MaxAttachmentsPerMessage`).

## Статусы сообщений

```
Sent (0) → Deleted (3)
```

`MessageStatus` также содержит значения `Delivered (1)` и `Read (2)`, но ни один код-путь в `Message` их не устанавливает — `Delivered`/`Read` не используются. Статус "прочитано" в этом мессенджере отслеживается не на уровне отдельного сообщения, а на уровне участника чата — `ChatMember.LastReadAt` (см. [Chats](chats.md), `MarkChatReadCommandHandler`): при открытии чата отмечается момент последнего прочтения, а не факт прочтения каждого сообщения по отдельности.

## Авторизация по членству в чате

Все эндпоинты (включая чтение истории) проверяют, что текущий пользователь **состоит в чате** — через `IChatMembershipChecker` (Shared.Kernel, реализация в [Chats](chats.md)). Если нет — `403 Forbidden`. Без этой проверки любой залогиненный пользователь, узнав GUID чужого чата, мог бы читать и писать в него; кикнутый из группы участник продолжал бы иметь доступ по старому `chatId`. У `forward` проверка двойная — и на чат-источник, и на чат-получатель (см. «Пересылка сообщений» ниже).

Каждое сообщение в ответе `GetMessages` дополнено `senderName`/`senderAvatarUrl` — резолвятся через `IUsersModule`, чтобы клиенту не приходилось показывать «обрезок UUID» вместо имени отправителя.

## Cursor-пагинация

Вместо постраничной загрузки используется **cursor** — ID последнего полученного сообщения:

```
Первый запрос: GET /messages?limit=50
  → возвращает 50 сообщений + nextCursor (ID 50-го)

Следующий запрос: GET /messages?before={nextCursor}&limit=50
  → возвращает следующие 50 сообщений
```

Это эффективнее страниц: не нужно считать OFFSET в БД.

Внутри курсор резолвится не по `sent_at`, а по монотонному `sequence` — несколько сообщений могут получить одинаковый `sent_at` (например, при пересылке пачки сообщений подряд), и сортировка по одному только времени могла бы пропустить или задвоить сообщение на границе страницы. Курсор (`before`) также ищется **строго внутри текущего чата** — раньше id сообщения-курсора резолвился без проверки `chatId`, и участник чата A мог передать id чужого сообщения из чата B (в котором не состоит) как курсор и по ответу узнать, что оно существует, и его точный `sent_at`.

## Удаление сообщения (мягкое)

`DELETE /api/chats/{chatId}/messages/{id}` → `DeleteMessageCommand` → `Message.Delete()`. Сообщения не удаляются физически. При удалении:
- `status = "Deleted"`
- `content = ""` (контент стирается)
- `deleted_at = NOW()`

Запись остаётся в БД для сохранения целостности истории (ответы и пересылки на удалённые сообщения продолжают на них ссылаться).

Удалить может **автор сообщения** или **Owner/Admin чата** (модерация) — не любой участник. Сначала обычная проверка членства (`IChatMembershipChecker.IsMemberAsync`, `403` если не состоишь в чате), затем: если запросивший не автор, проверяется `IChatMembershipChecker.CanModerateAsync` (роль Owner/Admin в `chats.members`, см. [Chats](chats.md)); не модератор и не автор — `403`. В личных чатах ролей выше `Member` не бывает, так что там это фактически всегда «только автор». Повторное удаление уже удалённого — доменная ошибка `Message.AlreadyDeleted`.

`DeleteMessagesCommand` (пакетное удаление) применяет то же правило поэлементно: сообщения, которые запросивший не вправе удалить, молча выпадают из батча — так же, как уже удалённые или из чужого чата (не 403 на весь запрос, а частичный успех, см. «Cursor-пагинация» о том же принципе идемпотентности выше).

> История: изначально удалять мог только автор. Затем это расширили до любого участника чата (в рамках задачи «удалить сообщение (любое)»). Итоговое правило — компромисс: рядовой участник управляет только своими сообщениями, а Owner/Admin сохраняют право модерации чужих — как в большинстве мессенджеров с ролями.

Редактирование и удаление (одиночное и пакетное) защищены optimistic concurrency через системную колонку Postgres `xmin` (см. схему выше) — если сообщение успели изменить/удалить параллельно между чтением и записью (например, два участника одновременно жмут "удалить" на одном сообщении), проигравший запрос получает `409 Conflict` вместо тихой перезаписи чужого изменения.

При успешном удалении поднимается `MessageDeletedDomainEvent` → Realtime-модуль рассылает `MessageDeleted` всем в группе `chat:{chatId}`, включая инициатора — это не проблема, так как у клиента пометка "удалено" идемпотентна.

**Фронтенд:** удаление вызывается из контекстного меню сообщения (правый клик). Пункт "Удалить" показывается только если сообщение своё или текущий пользователь — Owner/Admin группы (`useGroupMembers.ts → isModerator`, вычисляется из роли в `groupMembers`; в личных чатах всегда `false`, там условие сводится к `msg.own`) — сервер всё равно перепроверит право независимо от того, показала ли кнопку разметка. То же правило и для массового удаления в режиме выделения: кнопка активна, если выделены только свои сообщения либо пользователь модератор (`ChatWindow.tsx → canBulkDelete`). Удалённое сообщение **полностью убирается** из списка, а не заменяется плашкой "Сообщение удалено" — и по live-событию `MessageDeleted` (`useChatMessages.ts → handleDeletedMessage`, `filter`, а не пометка), и при обычной загрузке истории (`chatsApi.ts → fetchMessages` отфильтровывает `dto.status === 'deleted'` ещё на этапе маппинга).

## Редактирование сообщения

`PATCH /api/chats/{chatId}/messages/{id}` → `EditMessageCommandHandler` → `Message.Edit()`. В отличие от удаления, редактировать может **только автор** — `Message.Edit()` сверяет `senderId` с запросившим и возвращает доменную ошибку, если это не он.

Членство в чате при этом проверяется отдельным вызовом `IChatMembershipChecker` **до** `Message.Edit()`, а не полагается на то, что раз он автор — значит, состоит в чате: если автора с тех пор исключили из группового чата, редактировать сообщение в нём он больше не должен, хотя формально остаётся его автором.

Как и удаление, редактирование защищено optimistic concurrency через `xmin` (см. схему выше) — параллельная правка того же сообщения с двух вкладок даёт `409 Conflict` на втором запросе.

## Пересылка сообщений (Forward)

`POST /api/chats/{targetChatId}/messages/forward`, тело — `{ sourceChatId, messageIds: [...] }`. Можно переслать сразу несколько сообщений (мультивыделение на фронтенде) в один целевой чат.

`ForwardMessagesCommandHandler`:
1. Проверяет членство инициатора **в обоих** чатах — и в `sourceChatId` (откуда берём сообщения), и в `targetChatId` (куда пересылаем), через `IChatMembershipChecker`
2. Грузит исходные сообщения одним запросом (`IMessageRepository.GetByIdsAsync`), отбрасывает те, что не из `sourceChatId` или уже удалены
3. Сортирует по `SentAt` — чтобы порядок пересланных сообщений в целевом чате совпадал с их хронологией в исходном, а не с порядком id в запросе
4. Для каждого создаёт **новую независимую копию** через `Message.CreateForwarded(...)`: автор копии (`SenderId`) — тот, кто переслал, а не оригинальный автор; `ForwardedFromMessageId`/`ForwardedFromUserId` — только для подписи «Переслано от N» на клиенте. Если пересылают уже пересланное сообщение, `ForwardedFromUserId` берётся из оригинала (`original.ForwardedFromUserId ?? original.SenderId`) — подпись всегда указывает на самого первого автора, а не на промежуточного пересылающего, сколько бы раз сообщение ни переслали дальше
5. Копия доставляется как обычное сообщение через `MessageSentDomainEvent` → `ReceiveMessage`, просто с двумя дополнительными полями в payload

Копия ведёт себя как обычное сообщение: пересылающий может её отредактировать или удалить, это не связано с правами на оригинал.

**Фронтенд:** модалка выбора чата (`frontend/src/features/messenger/ForwardModal/`) со списком чатов пользователя; кнопка «Переслать» есть и в контекстном меню одного сообщения, и в панели мультивыделения. Пересланное сообщение показывает плашку «Переслано от {имя}» над текстом.

## Ответ на сообщение (Reply)

1. Цитируемое сообщение обязано быть из того же чата, что и новое — `SendMessageCommandHandler` явно это проверяет и возвращает `422 Validation`, если нет. Без этой проверки клиент мог бы подставить id сообщения из чужого чата, в котором даже не состоит, и получить в ответ его текст и автора через резолвленное reply-превью (см. ниже)
2. `MessageSentDomainEvent` несёт `ReplyToMessageId`
3. В реальном времени (`MessageSentEventHandler`) и в истории (`GetMessagesQueryHandler`) сервер одним батч-запросом резолвит **превью** цитируемого сообщения — имя автора и текст, обрезанный до 120 символов (`IMessagesModule.GetMessagePreviewsByIdsAsync`) — и кладёт их прямо в payload/DTO (`replyToSenderName`, `replyToContent`), чтобы клиенту не нужно было делать отдельный запрос или держать историю в памяти
4. Если исходное сообщение к этому моменту удалено — `replyToContent` приходит `null`, клиент показывает «Исходное сообщение удалено» вместо пустой цитаты

**Фронтенд:** «Ответить» в контекстном меню сообщения (доступно для любого — своего и чужого) открывает панель над полем ввода с именем автора и текстом цитируемого сообщения; при отправке ответ рендерится с цитатой над текстом.

## Прикреплённые файлы

Когда пользователь отправляет файл(ы):
1. `POST /api/chats/{id}/messages/upload` — один или несколько файлов (`multipart/form-data`) + опциональный `caption`
2. `UploadAndSendMessageCommandHandler` грузит файлы **последовательно** (не параллельно — Files-модуль пишет через тот же scoped `DbContext`, EF Core не поддерживает параллельные операции на одном инстансе), для каждого вызывая `IFilesModule.UploadChatAttachmentAsync()` (см. [Files](files.md))
3. Создаётся **одно** сообщение с несколькими `MessageAttachment` (см. схему выше) — максимум **10** вложений (`Message.MaxAttachmentsPerMessage`), `caption` относится ко всему сообщению целиком, а не к отдельному файлу
4. Если загрузка любого файла в батче или итоговое сохранение сообщения проваливается — уже загруженные до этого файлы этого же запроса удаляются компенсирующим вызовом `IFilesModule.DeleteChatAttachmentAsync()` (`try/finally`), чтобы не оставались в хранилище без владельца

Файл валидируется трижды: белый список MIME-типов (`AllowedAttachmentMimeTypes`, см. [Files](files.md)), лимит размера, и сверка по сигнатуре первых байт файла с заявленным `Content-Type` (`FileSignatureValidator`) — не даёт выдать, например, исполняемый файл за картинку простой подменой заголовка запроса.

## Поиск по сообщениям

`GET /api/chats/{chatId}/messages/search?q=...` → `SearchMessagesQueryHandler`. Ищет **по словам, а не по подстроке**: и запрос, и текст каждого сообщения чата разбиваются на слова регэкспом `[\p{L}\p{N}]+` (юникод — корректно работает и с кириллицей), сообщение считается совпадением только если **каждое** слово запроса — начало какого-то слова сообщения. Например, "поход" находит "поход"/"походы", но не "выход" — в отличие от обычного `ILIKE '%поход%'`, который заодно нашёл бы и "выход".

Сопоставление идёт не в SQL, а в памяти приложения: сообщения чата (не удалённые, `Kind = Text`) вытягиваются одним запросом (`IMessageRepository.GetSearchableByChatIdAsync`), дальше фильтрация — обычный LINQ. Это сознательный выбор ради предсказуемости — Postgres full-text search (`to_tsvector`/`plainto_tsquery`) дал бы то же самое через нативный `@@`-оператор и был бы быстрее на очень больших чатах, но его конфигурация языка (`'russian'` vs `'simple'`) и стемминг менее предсказуемы, чем прямое сравнение префиксов слов. Результат ограничен 100 сообщениями (`MaxResults`), отсортирован по `SentAt` (сначала новые).

**Фронтенд:** кнопка-лупа в правом углу шапки чата (`ChatWindow.tsx`) открывает `ChatSearchPanel` — поле ввода с debounce 300мс и список результатов. Найденное сообщение может быть в истории, ещё не подгруженной на клиент (поиск идёт по всей БД, а клиент держит только последние страницы) — по клику на результат `MessengerPage.tsx` не просто скроллит, а **дозагружает историю постранично** (переиспользуя обычный `loadMoreHistory`) до тех пор, пока целевое сообщение не окажется среди уже загруженных, и только тогда `ChatWindow` прокручивает к нему и подсвечивает — по тому же эффекту, которым уже отслеживается появление сообщения в `messages`.

## Опросы

Опрос — это сообщение с `message_type = "Poll"`, а не отдельная сущность верхнего уровня: `content` хранит **текст вопроса** (та же колонка, что и обычный текст сообщения), а варианты ответа и голоса — собственные owned-коллекции на агрегате `Message`, по образцу `Attachments`/`Reactions`.

### Таблица `messages.poll_option`

| Колонка | Тип | Описание |
|---|---|---|
| id | uuid | Первичный ключ |
| message_id | uuid | Владеющее сообщение-опрос (FK на messages.message, CASCADE) |
| text | varchar(100) | Текст варианта ответа |
| sort_order | int | Порядок вариантов, заданный при создании |

### Таблица `messages.poll_vote`

| Колонка | Тип | Описание |
|---|---|---|
| id | uuid | Первичный ключ |
| message_id | uuid | Сообщение-опрос (FK на messages.message, CASCADE) |
| option_id | uuid | За какой вариант — **не** формальный FK на `poll_option` (тот же стиль, что и `reply_to_message_id` и другие "слабые" ссылки в этом модуле), сверяется на уровне домена |
| user_id | uuid | Кто проголосовал |
| voted_at | timestamptz | Когда |

**Индекс:** `ux_poll_vote_message_id_user_id` (unique) — один голос на пользователя на опрос **на уровне БД**, а не только в коде: `Message.SetPollVote()` перед добавлением голоса всегда удаляет предыдущий голос этого же пользователя (если был), так что строк-дублей быть не должно, но уникальный индекс — гарантия на случай гонки/бага, а не просто документация намерения.

**Только групповые чаты.** `CreatePollCommandHandler` проверяет тип чата через `IChatMembershipChecker.IsGroupChatAsync` — этот метод специально добавлен в общий (Shared.Kernel) интерфейс членства, а не в `IChatsModule`: модуль Messages не может ссылаться на модуль Chats напрямую (это создало бы циклическую зависимость проектов, так как Chats уже ссылается на Messages), а `IChatMembershipChecker` — единственная общая точка, через которую Messages может спросить что-либо о чате, не зная о Chats как о модуле.

**Голосование** (`SetPollVoteCommand` → `Message.SetPollVote(userId, optionId)`) — один голос на пользователя, повторный вызов с другим `optionId` меняет голос, а `optionId: null` (`DELETE .../vote`) — отменяет, ровно как `SetReaction` уже делает для реакций.

**Голосующий не анонимен** — задача явно требовала видеть, кто за что проголосовал (в отличие от анонимных опросов Telegram по умолчанию). `GetMessagesQueryHandler` резолвит для каждого варианта полный список проголосовавших (`PollOptionDto.Voters` — id/имя/аватар через `IUsersModule`, тот же батч-запрос, что уже используется для отправителей/реакций), а не просто счётчик.

**Реалтайм.** Голос/отмена поднимает `PollVoteChangedDomainEvent` → Realtime-модуль рассылает `PollVoteChanged` всем в группе `chat:{chatId}`, включая само голосовавшее соединение (см. [Realtime](realtime.md)) — фронтенд не делает оптимистичное обновление своего голоса локально, а ждёт этого же события, что и для реакций. Только что созданный опрос доставляется как обычное `ReceiveMessage` с дополнительным полем `pollOptions` (варианты без голосов — они не могут появиться раньше самого сообщения).

**Обновление профиля голосующего.** Если пользователь меняет имя/аватар, `handleUserProfileUpdated` на фронтенде патчит не только отправителей сообщений, но и записи в `poll.options[].voters[]` во всех уже загруженных опросах — иначе имя голосовавшего осталось бы замороженным на момент голосования до следующей полной перезагрузки истории.

**Фронтенд:** кнопка-скрепка в **группах** при наведении курсора показывает меню "Отправить файл"/"Создать опрос" (в личных чатах меню нет вообще — бэкенд всё равно откажет в создании опроса, но UI не показывает недоступный пункт даже как соблазн). Закрытие меню намеренно с задержкой ~150мс после ухода курсора (`ChatWindow.tsx → closeAttachMenu`) — без неё курсор успевал покинуть кнопку раньше, чем добирался до самого меню (зазор между ними), и меню закрывалось не открывшись толком. Клик по варианту голосует; клик по уже выбранному — отменяет голос; по правому клику на опрос пункт "Отменить голос" доступен предпоследним, перед "Удалить сообщение". Список проголосовавших под каждым вариантом сворачиваемый — по умолчанию видны только аватарки (до 5) и счётчик, полные имена — по клику; клик по имени открывает профиль этого пользователя.

> Отдельная деталь, не связанная с самими опросами, но найденная при их отладке: после выбора файлов через системный диалог фокус остаётся на самой кнопке-скрепке (`<input type="file">` скрыт через `display:none` и фокус в принципе принять не может) — нажатие Enter в этот момент срабатывало как повторный клик по кнопке (стандартное поведение браузера для сфокусированной `<button>`), заново открывая проводник вместо отправки сообщения. Фикс — явный `focus()` на текстовое поле сразу после выбора файлов.

## Системные сообщения

Изменения состава группового чата (добавили участника / кто-то вышел сам / кого-то удалили) попадают в ленту сообщений как обычная запись с `message_type = "System"`, а не отдельным событием мимо истории — так лента остаётся единым источником правды о происходившем в чате, без отдельного API для получения «истории событий».

`Chats`-модуль вызывает `IMessagesModule.CreateSystemMessageAsync(chatId, actorUserId, targetUserId, eventType)` из `AddChatMemberCommandHandler`/`RemoveChatMemberCommandHandler` → `Message.CreateSystem()`. `actorUserId` — кто выполнил действие, `targetUserId` — кого оно коснулось; при самостоятельном выходе это один и тот же пользователь (`SystemEventType.MemberLeft` вместо `MemberRemoved` определяется в `RemoveChatMemberCommandHandler` сравнением `requesterId == userId`).

`content` для системного сообщения — технический маркер вида `"system:MemberAdded"`, а не текст для показа: он не локализован и не предназначен для отображения. Клиент строит текст сам из `messageType`/`systemEventType`/`targetUserId` (резолвя имя по `targetUserId` так же, как имя обычного отправителя), поэтому системные сообщения не переводятся сервером и не всплывают как push-уведомления (модуль Notifications их явно отфильтровывает при рассылке).

## Доменные события (Domain Events)

Когда сообщение создаётся, редактируется или удаляется, в объекте `Message` поднимается доменное событие:

- `MessageSentDomainEvent` — при создании (обычном, файловом, пересылке или опросе). Несёт `ForwardedFromMessageId`/`ForwardedFromUserId` (заполнены только для пересланных копий), `ReplyToMessageId` (заполнен для ответов) и `PollOptions` (заполнен только для опросов) — Realtime-модуль резолвит по ним имена/превью перед рассылкой (см. «Пересылка», «Ответ на сообщение», «Опросы» выше и [Realtime](realtime.md))
- `MessageEditedDomainEvent` — при редактировании
- `MessageDeletedDomainEvent` — при удалении (несёт только `messageId`/`chatId` — содержимое и так уже стёрто, рассылать в WebSocket нечего)
- `MessageReactionChangedDomainEvent` — при простановке/снятии реакции
- `PollVoteChangedDomainEvent` — при голосовании/отмене голоса в опросе (см. «Опросы» выше)

После `SaveChanges()` в `MessagesDbContext` они автоматически публикуются через MediatR — и их подхватывает Realtime-модуль.
